오픈칼로리 다운로드
  • 개발 동기
    하나의 프로젝트를 처음부터 끝까지 온전히 내 손으로 만들어 보고 싶다는 목표에서 시작하여 실 사용자가 있는 '케이크 칼로리 비교 앱'을 개발했습니다. 단순한 칼로리 비교를 넘어 사용자들이 쉽고 재미있게 정보를 접할 수 있도록 UI/UX를 설계했으며, 특히 사용자 경험을 고려한 데이터 필터링 및 정렬 기능을 구현하며 개발 역량을 키웠습니다. 기획부터 분석, 설계, 개발, 배포, 유지보수까지 전 과정을 직접 경험하며, 서비스 기획과 기술적 문제 해결 능력을 동시에 향상시킬 수 있었습니다.
  • UI / UX
    사용자의 건의사항 건의 내용: 어떤 케이크가 더 적은 칼로리인지 한눈에 알 수 있었으면 좋을 것 같아요.
    개선 사항: 케이크 사진 좌측 상단에 ☆low 문자를 통해 사용자에게 직관적인 화면을 제공했습니다.
    제스처 인식 오류 해결
    원인: 케이크 이미지 선택 시 탭 제스처를 활용한 세그웨이 전환이 동작하지 않는 문제가 발생했습니다.
    개선 방법: 공식 문서를 참고하여, 겹쳐진 뷰에서는 하나의 탭제스처만 인식되도록 설계되어 있음을 확인
    했습니다. 이를 해결하기 위해 중첩된 뷰의 터치 권한을 제거하여 제스처가 정상적으로 동작하도록 수정
    했습니다.
  • 데이터 관리
    - Realm을 사용한 기본적인 로컬 데이터 CRUD 구현했습나다.
    *변경점: db관리와 운용을 파이어베이스 Realtime DB와 Storage로 바꿨습니다.
    원인과 해결방법: 초기에는 Core Data를 사용하여 로컬 데이터 관리를 구현했으나, 배포 과정에서 실시간 동기화의 필요성을 인지하고 빠르게 Firebase Realtime DB와 Storage로 전환을 결정했습니다. 이 과정에서 데이터 마이그레이션 및 구조 변경으로 인해 예상 대비 개발 시간이 2배 증가했지만, 사전 설계의 중요성을 경험하며, 이후 DB를 사용하는 코드의 개발 시간을 50% 절감할 수 있었습니다.
  • 화면 간 데이터 전달 문제와 구조 개선
    문제: ViewController 간 데이터 전달 시, 새로 생성된 인스턴스에서 데이터를 변경해도 기존 화면에는 반영되지 않는 문제가 발생했습니다.
    개선 방법: 클로저를 활용하여 화면간 데이터 전송을 구현했습니다. 또한, weak self를 사용하여 메모리 누수를 방지하고 안전한 데이터 전달을 구현했습니다.
    문제: 코드 리팩토링 후, ViewController에서 변경 사항을 감지할 수 없어 UI가 자동으로 업데이트되지 않는 구조였습니다.
    개선 방법: Combine을 사용하여 파이어베이스 데이터 프로퍼티의 변경을 실시간으로 감지하도록 설정했습니다. 데이터 변경을 감지할 시 자동으로 테이블뷰를 업데이트 하도록 구현했습니다.
  • 클린 아키텍쳐와 유지보수
    문제: 앱을 개발후에 클린 아키텍처 관점에서 코드를 점검한 결과, 모델과 뷰 컨트롤러의 역할이 혼재되어 유지보수가 어려운 구조임을 확인했습니다.
    개선 사항:
    • 네트워크를 사용하는 Firebase 로직을 ViewModel로 분리하여 데이터 처리와 UI 로직을 분리했습니다.
    • 데이터 필터링 옵션 및 데이터 엔티티를 Model로 구성하여 책임을 분배했습니다.
    • UI 관련 코드를 ViewController에서만 처리하여 역할을 명확히 구분했습니다.
    • Combine을 적극 활용하여 비동기 데이터 스트림을 관리하고, UI 업데이트를 실시간으로 반영했습니다.
    • 결론: 코드 구조가 명확해지면서 기능 추가 및 유지보수 시간이 줄어들었고, 개발 생산성이 향상되었습니다.
